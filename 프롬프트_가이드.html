1. 개요

1.1 이 가이드의 목적

이 문서는 AI(Claude)를 활용하여 전략 보고서를 작성한 전체 프로세스를 정리한 것이다. 단순히 "AI에게 보고서 써줘"라고 요청하는 것이 아니라, 에이전트 팀 아키텍처 설계 → 병렬 리서치 → 파이프라인 실행 → 피드백 시뮬레이션 → 반복 개선 → 포맷 전환이라는 체계적 워크플로우를 통해 높은 품질의 보고서를 산출하는 방법을 공유한다.

특히, 이 가이드는 각 단계에서 실제로 사용된 프롬프트 원문을 모두 포함한다. 요약이 아닌 실제 텍스트를 제공하여 재현성을 극대화한다.

3. 실제 사용 프롬프트 원문

이 섹션은 프로젝트에서 실제로 에이전트에게 전달된 프롬프트 원문을 8개 카테고리로 정리한 것이다.

3-1. 리서치 에이전트 프롬프트 (Phase 1)

3개 에이전트를 병렬로 실행한다. 각 프롬프트의 핵심은 조사 목적, 세부 항목(검색 키워드 포함), 출력 경로를 명확히 하는 것이다.

Research Agent 1 -- 네이버 마이팀 현황 & 국내 비교 (원문)
왜 이렇게 썼는가: 조사 목적("내부 전략팀을 위한 보고서")을 먼저 명시하여 에이전트가 조사 깊이와 톤을 조절하게 한다. 세부 항목에 "알림, 일정 표시, 구독 UX"처럼 기능 단위로 나열하여 누락을 방지한다.
당신은 보고서 작성을 위한 리서치 에이전트입니다. 아래 주제에 대해 깊이 있는 조사를 수행하고 결과를 파일로 저장하세요.

## 조사 주제: 네이버 스포츠 마이팀 현황 분석 + 국내 유사 서비스 비교
## 조사 목적: 내부 전략팀을 위한 보고서 작성. 네이버 스포츠 마이팀의 구독자 수 증가 및 서비스 개선 방안 도출을 위한 현황 파악.
## 세부 조사 항목:
1. 네이버 스포츠 마이팀 서비스 기능 전수 조사 (알림, 일정 표시, 구독 UX 등 상세)
2. 국내 유사 서비스: 카카오스포츠, 다음스포츠, SPOTV, 네이버 내 다른 구독 서비스
3. 사용자 반응(앱리뷰, 커뮤니티)

3-2. 파이프라인 오케스트레이션 프롬프트

리서치 결과를 하나로 병합한 뒤, outline → writer → fact-check → editor 순차 실행을 오케스트레이터에게 위임한다.

병합 + 파이프라인 순차 실행 프롬프트 (원문)
왜 이렇게 썼는가: 6개 Step을 한 프롬프트에 순서대로 나열하여 오케스트레이터가 전체 흐름을 한 번에 파악하고 각 단계의 의존성을 이해하게 한다. 각 Step에서 "어떤 에이전트 역할을 수행하라"고 명시하여 역할 전환을 명확히 한다.
당신은 보고서 작성 파이프라인의 오케스트레이터입니다. 아래 작업을 순서대로 수행하세요.

Step 1: 리서치 병합 (팬인) -- 3개 파일 → 01_research.md
Step 2: 목차 설계 (outline-agent 역할) -- outline-agent.md 읽고 역할 수행
Step 3: 초안 작성 (writer-agent 역할)
Step 4: 사실 검증 (fact-check-agent 역할)
Step 5: 최종 편집 (editor-agent 역할)
Step 6: 최종 파일 복사

CPM 피드백 시뮬레이션 -- 사용자 입력 원문
왜 이렇게 썼는가: "귀신같은 분", "꼼꼼하고 날카롭지", "유저 우위" 같은 구어체 묘사가 페르소나의 성격을 AI에게 효과적으로 전달한다. 격식체 프롬프트보다 오히려 이런 자연어 묘사가 더 정확한 피드백을 이끌어낸다.
결과물에 대해 Chief Product Manager가 해당 결과물에 대해 어떤 평가 피드백을 할 것같어?
피드백과 예상 질문을 뽑아볼 수 있을까?
Chief Product Managers는 네이버 서비스를 통달한 귀신같은 분이고 상당히 꼼꼼하고 날카롭지..
무엇보다 유저 우위로 유저들에게 서비스가 어떤 benefit을 줄 수 있는지도 평가하는 분이야.

5-3. 피드백 시뮬레이션의 위력

  • 관심 KPI (DAU, 매출, 리텐션 등)
  • 의사결정 성향 ("데이터 없으면 안 됨" 등)

같은 내용이라도 독자에 따라 최적의 포맷이 다르다.

  • Before/After 예시를 프롬프트에 포함하라: 추상적 지시보다 구체적 변환 예시가 효과적
  • 회의 어젠다 형태를 활용하라: "오늘 결정할 것 3가지"는 회의 참석자의 집중을 유도
  • [논의 포인트] 삽입: 독자가 수동적으로 읽는 것이 아니라 능동적으로 참여하게 만듦

5-7. 루브릭 자체 평가를 포함하라

  • 투명성: 보고서의 한계를 독자가 바로 파악
  • 신뢰도: "이 보고서가 무엇을 모르는지"를 알려주므로 신뢰가 올라감
  • 후속 액션: "완전 5점을 위해 필요한 것"이 명시되어 다음 스텝이 명확

6. 사용된 에이전트/스킬 목록

에이전트 역할 요약

에이전트역할스킬
research-agent주제별 웹 리서치 수행. 검색 키워드 기반으로 데이터를 수집하고 구조화된 리서치 문서를 산출report-research
outline-agent리서치 결과를 기반으로 보고서의 목차, 구조, 섹션별 핵심 메시지를 설계report-outline
writer-agent구조 설계안과 리서치 자료를 기반으로 본문 초안을 작성report-draft
fact-check-agent초안의 모든 수치와 주장을 리서치 원본과 대조 검증(내장 검증 로직)
editor-agent문체 통일, 흐름 개선, 포맷 정리, 최종 품질 관리report-edit

하네스(Harness) 스킬의 역할

하네스는 에이전트 팀의 오케스트레이터다.

  1. 에이전트 팀 구성: 작업에 필요한 에이전트들을 정의하고 각각에 스킬을 배정
  2. 실행 순서 관리: 팬아웃(병렬)/파이프라인(순차) 실행 흐름을 제어
  3. 파일 경로 연결: 각 에이전트의 출력을 다음 에이전트의 입력으로 전달
  4. 품질 게이트: 각 단계 완료 후 산출물 품질을 확인하고 다음 단계 진행 여부 판단

이 프로젝트에서는 report-orchestrator 스킬이 하네스 역할을 수행했다.

7-2. 변경하지 않아도 되는 범용 패턴

  1. 4-Phase 반복 개선 구조: v1(빠른 초안) → v2(피드백 반영) → v3(타겟 보강) → v4(포맷 전환)
  2. 팬아웃/팬인 + 파이프라인 아키텍처
  3. 파이프라인 4단계: outline → writer → fact-check → editor
  4. 피드백 시뮬레이션 패턴: 페르소나 → 비판적 검토 → 개선 방향
  5. 검색 키워드 명시: 리서치 프롬프트에 구체적 검색 쿼리 포함
  6. [검증 필요] 태깅: 확인 불가 정보는 명시적으로 표시
  7. 루브릭 자체 평가: 보고서 말미에 평가 매트릭스 포함
  8. 두괄식 구조: 섹션 첫 문장 = 핵심 메시지
  9. 포맷 전환: 같은 내용을 독자/상황에 맞게 재구성

7-3. 실행 체크리스트

  • 요구사항 정의: 보고서 목적, 독자, 분량 결정
  • 리서치 영역 3개 정의: 주제에 맞는 조사 영역 분류
  • 리서치 에이전트 3개 병렬 실행: 검색 키워드 명시
  • 리서치 병합 (팬인)
  • 파이프라인 실행: outline → writer → fact-check → editor
  • v1 산출
  • 피드백 시뮬레이션: 실제 리뷰어 페르소나로 비판적 검토
  • 보강 리서치 병렬 실행: 피드백에서 도출된 약점 영역
  • v2 재작성
  • 루브릭 갭 분석: 각 평가 축별 현재 점수 vs 목표
  • 타겟 리서치 병렬 실행: 갭을 메울 데이터 수집
  • v3 최종 재작성: 보강 리서치 + 구조 재정의
  • 루브릭 자체 평가 포함
  • 독자/상황에 맞는 포맷 전환 (v4): 회의 자료, 발표 자료 등
