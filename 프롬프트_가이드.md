> **프로젝트:** 네이버 스포츠 마이팀 전략 보고서  
> **최종 산출물:** v4 (조직장 리뷰 회의 자료)  
> **작성 도구:** Claude Code + 에이전트 하네스

## 1. 개요

### 1.1 이 가이드의 목적

이 문서는 AI(Claude)를 활용하여 전략 보고서를 작성한 전체 프로세스를 정리한 것이다. 단순히 "AI에게 보고서 써줘"라고 요청하는 것이 아니라, **에이전트 팀 아키텍처 설계 → 병렬 리서치 → 파이프라인 실행 → 피드백 시뮬레이션 → 반복 개선 → 포맷 전환**이라는 체계적 워크플로우를 통해 높은 품질의 보고서를 산출하는 방법을 공유한다.

특히, 이 가이드는 각 단계에서 **실제로 사용된 프롬프트 원문**을 모두 포함한다. 요약이 아닌 실제 텍스트를 제공하여 재현성을 극대화한다.

### 1.2 최종 산출물

- **보고서명:** 네이버 스포츠 마이팀 전략 보고서 v4 (조직장 리뷰 회의 자료)
- **루브릭 점수:** v3 기준 39.5 / 40 (98.8%)
- **평가 8개 축:** 유저 인사이트, 데이터 근거, 실행 가능성, 비즈니스 임팩트, 전략적 논리, 유저 밸류, 경쟁 분석, 실행 로드맵
- **점수 변화:** v1(~18/40) → v2(29.5/40) → v3(39.5/40) → v4(포맷 전환: 회의 자료)

## 3. 실제 사용 프롬프트 원문

이 섹션은 프로젝트에서 **실제로 에이전트에게 전달된 프롬프트 원문**을 카테고리별로 정리한 것이다. 요약이 아닌 실제 텍스트를 포함하여 재현성을 극대화한다.

### 3-1. 리서치 에이전트 프롬프트 (Phase 1)

리서치 단계에서는 3개 에이전트를 병렬로 실행한다. 각 프롬프트의 핵심은 **조사 목적, 세부 항목(검색 키워드 포함), 출력 경로**를 명확히 하는 것이다.

<details>
<summary>Research Agent 1 -- 네이버 마이팀 현황 & 국내 비교 (원문)</summary>

> **왜 이렇게 썼는가:** 조사 목적("내부 전략팀을 위한 보고서")을 먼저 명시하여 에이전트가 조사 깊이와 톤을 조절하게 한다. 세부 항목에 "알림, 일정 표시, 구독 UX"처럼 기능 단위로 나열하여 누락을 방지한다.

```
당신은 보고서 작성을 위한 리서치 에이전트입니다. 아래 주제에 대해 깊이 있는 조사를 수행하고 결과를 파일로 저장하세요.

## 조사 주제: 네이버 스포츠 마이팀 현황 분석 + 국내 유사 서비스 비교
## 조사 목적: 내부 전략팀을 위한 보고서 작성. 네이버 스포츠 마이팀의 구독자 수 증가 및 서비스 개선 방안 도출을 위한 현황 파악.
## 세부 조사 항목:
1. 네이버 스포츠 마이팀 서비스 기능 전수 조사 (알림, 일정 표시, 구독 UX 등 상세)
2. 국내 유사 서비스: 카카오스포츠, 다음스포츠, SPOTV, 네이버 내 다른 구독 서비스
3. 사용자 반응(앱리뷰, 커뮤니티)
## 출력: _workspace/01a_research_naver_domestic.md (파일 구조 지정)
```

</details>

5. 중장기 트렌드 (AI, 소셜, 굿즈/티켓)

### 3-2. 파이프라인 오케스트레이션 프롬프트

리서치 결과를 하나로 병합한 뒤, outline → writer → fact-check → editor 순차 실행을 오케스트레이터 에이전트에게 위임한다.

<details>
<summary>병합 + 파이프라인 순차 실행 프롬프트 (원문)</summary>

> **왜 이렇게 썼는가:** 6개 Step을 한 프롬프트에 순서대로 나열하여 오케스트레이터가 전체 흐름을 한 번에 파악하고 각 단계의 의존성을 이해하게 한다. 각 Step에서 "어떤 에이전트 역할을 수행하라"고 명시하여 역할 전환을 명확히 한다.

```
당신은 보고서 작성 파이프라인의 오케스트레이터입니다. 아래 작업을 순서대로 수행하세요.

Step 1: 리서치 병합 (팬인) -- 3개 파일 → 01_research.md
Step 2: 목차 설계 (outline-agent 역할) -- outline-agent.md 읽고 역할 수행
Step 3: 초안 작성 (writer-agent 역할)
Step 4: 사실 검증 (fact-check-agent 역할)
Step 5: 최종 편집 (editor-agent 역할)
Step 6: 최종 파일 복사
```

</details>

### 3-3. CPM 피드백 시뮬레이션 프롬프트 (Phase 2)

이 패턴이 v1에서 v2로의 도약을 만든 핵심이다. 실제 리뷰어의 성향까지 구체적으로 묘사한다.

<details>
<summary>CPM 피드백 시뮬레이션 -- 사용자 입력 원문</summary>

> **왜 이렇게 썼는가:** "귀신같은 분", "꼼꼼하고 날카롭지", "유저 우위" 같은 구어체 묘사가 페르소나의 성격을 AI에게 효과적으로 전달한다. 격식체 프롬프트보다 오히려 이런 자연어 묘사가 더 정확한 피드백을 이끌어낸다.

```
결과물에 대해 Chief Product Manager가 해당 결과물에 대해 어떤 평가 피드백을 할 것같어?
피드백과 예상 질문을 뽑아볼 수 있을까?
Chief Product Managers는 네이버 서비스를 통달한 귀신같은 분이고 상당히 꼼꼼하고 날카롭지..
무엇보다 유저 우위로 유저들에게 서비스가 어떤 benefit을 줄 수 있는지도 평가하는 분이야.
```

</details>

**CPM 피드백 시뮬레이션에서 나온 7개 피드백 항목(실제 결과):**

1. "구독자 500만" 목표의 근거가 희박 → **수익화 모델 & 목표 역산 리서치** 추가
2. 유저 세그먼트 분석 부재 → **Avid/Casual/Light 3-Tier 분석** 추가
3. [검증 필요] 태그 3건 미해결 → **실제 웹 검색으로 팩트체크** 수행

7. "왜 지금인가"의 긴급성 논증 부재 → **경쟁 위협 타임라인** 신설

## 4. 핵심 프롬프트 패턴

### 4-1. 에이전트 팀 설계 프롬프트

에이전트 팀 아키텍처는 **팬아웃/팬인(Fan-out/Fan-in) + 파이프라인(Pipeline)** 하이브리드 구조를 사용한다.

**팬아웃/팬인이란?**
- **팬아웃:** 하나의 작업을 여러 에이전트에게 동시에 분배
- **팬인:** 여러 에이전트의 결과를 하나로 합침

**왜 3개 리서치를 병렬로 실행하는가?**
1. **시간 단축:** 3개를 순차로 하면 30분, 병렬로 하면 10분
2. **관점 분리:** 각 에이전트가 다른 영역에 집중하여 깊이 확보
3. **컨텍스트 독립:** 서로 다른 검색을 하므로 간섭 없이 품질 유지

**리서치 에이전트에게 주는 프롬프트의 핵심 요소:**

```
리서치 프롬프트 = {
  1. 조사 목적: "왜 이 조사를 하는가" (보고서 전체 맥락)
  2. 세부 항목: 조사할 영역 목록 (검색 키워드 포함)
  3. 출력 경로: 결과를 저장할 파일 경로
  4. 파일 구조: 섹션별 구조, 포함할 데이터 형식
  5. 품질 기준: "최소 3개 출처", "[지식 기반, 검색 불가] 표시" 등
}
```

### 4-2. 리서치 프롬프트 패턴

리서치 에이전트에게 주는 프롬프트의 핵심은 **구체적인 검색 키워드를 명시**하는 것이다.

**나쁜 예:**
```
네이버 스포츠 마이팀에 대해 조사해줘.
```

**좋은 예:**
```
# 리서치 주제: 네이버 스포츠 마이팀 현황 및 국내 비교

## 목적
내부 전략팀 보고서 -- 마이팀 구독자 수 증가 및 서비스 개선 방안 도출

## 세부 항목
1. 마이팀 핵심 기능 및 수치
   - 검색: "네이버 스포츠 마이팀 이용자 수 2026"
   - 검색: "네이버 마이티켓 발급"
   - 검색: "네이버 오픈톡 방문자 KBO"
2. 지원 종목 현황
   - 검색: "네이버 스포츠 마이팀 종목 지원"
   - 각 종목별 마이팀 지원 여부 확인
3. 국내 경쟁 서비스 비교
   - 검색: "다음스포츠 팀 팔로우"
   - 검색: "쿠팡플레이 스포츠 기능"
   - 검색: "SPOTV NOW 팀 구독"

## 출력 파일
`_workspace/01a_research_naver_domestic.md`

## 품질 기준
- 모든 수치에 출처(URL) 명시
- 검색으로 확인 불가 시 [지식 기반, 검색 불가] 표시
- 최소 3개 독립 출처에서 교차 확인
```

**검색 키워드를 명시하는 것이 중요한 이유:**
- AI가 "적절한" 검색어를 추측하지 않고, 의도한 데이터를 정확히 찾음
- 검색 결과의 재현성이 높아짐
- 조사 범위가 명확해져 누락이 줄어듦

**"[지식 기반, 검색 불가]" 표시 규칙:**
- 웹 검색으로 확인할 수 없는 정보는 반드시 이 태그를 붙임
- 이후 팩트체크 단계에서 이 태그가 붙은 항목을 우선 검증
- 보고서에 근거 없는 주장이 포함되는 것을 방지

### 4-3. 파이프라인 프롬프트 (outline → writer → fact-check → editor)

파이프라인은 4단계 순차 실행이며, 각 단계의 출력이 다음 단계의 입력이 된다.

```
[리서치 결과] ──→ outline-agent ──→ [구조 설계]
                                        │
                                        ▼
[리서치 결과] + [구조 설계] ──→ writer-agent ──→ [초안]
                                                    │
                                                    ▼
[초안] + [리서치 결과] ──→ fact-check-agent ──→ [검증 보고서 + 수정된 초안]
                                                    │
                                                    ▼
[수정된 초안] + [검증 보고서] ──→ editor-agent ──→ [최종본]
```

**각 에이전트별 핵심 지시:**

| 에이전트 | 핵심 지시 | 산출물 |
|---------|----------|--------|
| **outline-agent** | "문서 유형(현황분석+전략제안), 독자(내부 전략팀), 분량(A4 15~25페이지)을 고려하여 목차와 각 섹션의 핵심 메시지를 설계하라" | 목차 + 섹션별 핵심 메시지 + 포함 데이터 목록 |
| **writer-agent** | "outline의 구조를 그대로 따르되, 리서치 데이터를 출처와 함께 인용하라. 각 섹션의 첫 문장이 핵심 메시지(두괄식)가 되게 하라" | 마크다운 초안 |
| **fact-check-agent** | "초안의 모든 수치/주장을 리서치 원본과 대조하라. 일치하면 PASS, 불일치하면 수정, 확인 불가하면 [검증 필요] 태그를 남겨라" | 검증 리포트 + 수정된 초안 |
| **editor-agent** | "문체 통일, 흐름 개선, 표/포맷 정리, 미해결 사항 처리를 수행하라. 현황분석은 객관적 서술체, 전략제안은 설득적 제안체로 통일" | 최종본 + 편집 노트 |

### 4-4. CPM 피드백 시뮬레이션 패턴

**프롬프트 구조:**

```
당신은 네이버 서비스기획 CPM(Chief Product Manager)입니다.

[페르소나]
- 10년차 이상 프로덕트 매니저
- KPI: DAU, 리텐션, ARPU
- 관심사: 실행 가능성, 수익화 모델, 유저 세그먼트
- 성향: 데이터 없는 주장에 대해 매우 엄격, "그래서 얼마?"라고 반드시 물음

[태스크]
아래 보고서를 CPM 관점에서 비판적으로 검토하고, 구체적인 피드백을 제공하세요.

[피드백 형식]
각 피드백 항목:
1. 문제점 (어디가, 무엇이 부족한지)
2. 이유 (왜 이것이 문제인지 -- CPM 관점)
3. 개선 방향 (구체적으로 어떤 데이터/분석이 필요한지)

[보고서]
(v1 보고서 전문)
```

**페르소나 설정이 중요한 이유:**
- "어떤 사람이 읽는가"를 명확히 하면 AI가 해당 관점의 약점을 정확히 짚음
- 일반적 피드백("더 구체적으로")이 아닌 역할 특화 피드백("수익 모델이 없다", "유저 세그먼트가 빠졌다")이 나옴
- 실제 리뷰 미팅 전에 사전 약점 보완 가능

**피드백을 다음 반복의 입력으로 사용하는 방법:**
- 피드백 항목별로 "해결에 필요한 리서치"를 매핑
- 각 리서치를 독립 에이전트로 병렬 실행
- 리서치 결과 + 피드백 목록을 재작성 프롬프트에 포함

### 4-5. 포맷 전환 패턴 (v4)

v4에서 도입된 새로운 패턴으로, **같은 내용을 다른 독자/형태로 전환**하는 기법이다.

**핵심 원칙:**

| 전략 문서 (v3) | 회의 자료 (v4) |
|---------------|---------------|
| "Avid 팬 전환율 5x" | "열성 팬은 일반 팬보다 유료 서비스를 5배 더 씁니다" |
| "Phase 1 Quick Win" | "우선 할 일 -- 개발 2주면 되는 것부터" |
| 전문 용어 | 모든 용어에 괄호 풀이 |
| 데이터 나열 | 숫자마다 "이 숫자가 의미하는 것" 설명 |
| 결론 제시 | [논의 포인트] / [여기서 결정할 것] 삽입 |
| 보고서 형태 | 회의 어젠다: "오늘 결정할 것 3가지"로 시작 |

**포맷 전환의 효과:**
- 의사결정자가 "이해하는 데 걸리는 시간"을 최소화
- 회의에서 바로 논의와 결정이 가능한 구조
- 실제 유저 발언 인용으로 공감 유도

## 5. 핵심 교훈 & 팁

### 5-1. 반복 개선이 핵심이다

| 버전 | 루브릭 점수 | 핵심 개선 |
|------|-----------|----------|
| v1 | ~18/40 | 기본 구조 + 글로벌 벤치마크 |
| v2 | 29.5/40 | CPM 피드백 반영 + 유저 세그먼트 + 수익화 모델 |
| v3 | 39.5/40 | 실데이터 전환 + 긴급성 논증 + 실행 설계 |
| v4 | 포맷 전환 | 전략 문서 → 조직장 리뷰 회의 자료 |

한 번에 완벽한 보고서를 만들려 하지 말 것. **빠르게 v1을 만든 뒤, 피드백 루프를 돌려 개선하는 것이 압도적으로 효율적이다.** v1을 만드는 데 가장 오래 걸리지만, v2와 v3는 v1의 절반 이하 시간에 완성된다. v4는 내용이 아닌 포맷만 바꾸는 것이므로 가장 빠르다.

### 5-2. 병렬 실행으로 속도를 확보하라

3개 리서치 에이전트를 동시에 실행하면 시간이 1/3로 단축된다. 리서치 에이전트 간에는 의존성이 없으므로 완전한 병렬 실행이 가능하다.

```
순차 실행:  [Agent 1: 10분] → [Agent 2: 10분] → [Agent 3: 10분] = 30분
병렬 실행:  [Agent 1: 10분]
            [Agent 2: 10분]  = 10분 (3배 빠름)
            [Agent 3: 10분]
```

### 5-3. 피드백 시뮬레이션의 위력

실제 리뷰어의 관점을 시뮬레이션하면 사전에 약점을 보완할 수 있다. 이 프로젝트에서 CPM 피드백 시뮬레이션이 **v1→v2에서 +11.5점 상승**의 가장 큰 동인이었다.

핵심은 페르소나를 구체적으로 설정하는 것이다:
- 직급/역할 (CPM, 사업부장 등)
- 관심 KPI (DAU, 매출, 리텐션 등)
- 의사결정 성향 ("데이터 없으면 안 됨" 등)

### 5-4. "추정"을 "근거"로 전환하라

v2에서 v3로의 +10점 상승은 대부분 **글로벌 벤치마크 추정치를 실제 데이터로 교체**한 것에서 비롯되었다.

| v2 (추정) | v3 (근거) |
|----------|----------|
| "알림 CTR은 업계 평균 수준으로 추정" | "Pushwoosh 2025 데이터: 알림 CTR 업계 평균 2.25%, 개인화 시 8%" |
| "여성 팬 비율 증가 추세" | "한국프로스포츠협회 n=35,000: 2030 여성 관람 비율 58.7%" |
| "쿠팡플레이가 위협" | "EPL 6년 4,200억 독점 + NBA/F1/K리그 + 48개 리그, 스포츠패스 월 9,900원" |

실제 VOC(디시인사이드, 클리앙, Threads), IR 데이터, 커뮤니티 목소리가 보고서의 설득력을 만든다.

### 5-5. 두괄식 구조를 지켜라

Manager를 위한 보고서는 결론부터. **각 섹션의 첫 문장이 핵심 메시지**가 되어야 한다.

```
# 좋은 예 (두괄식)
## 1. 왜 지금인가
6개월 후면 돌이킬 수 없다. 쿠팡플레이/치지직/티빙이 동시에 팬 생태계를
완성하는 2026 하반기~2027 상반기가 마이팀의 마지막 선점 기회다.

# 나쁜 예 (미괄식)
## 1. 시장 환경 분석
최근 스포츠 산업에서는 다양한 변화가 일어나고 있다. OTT 플랫폼의 성장,
스포츠 중계권 가격 상승, MZ세대의 새로운 소비 패턴 등... (한참 뒤에 결론)
```

### 5-6. 루브릭 자체 평가를 포함하라

보고서에 자체 평가 루브릭을 포함하면:
- **투명성:** 보고서의 한계를 독자가 바로 파악
- **신뢰도:** "이 보고서가 무엇을 모르는지"를 알려주므로 신뢰가 올라감
- **후속 액션:** "완전 5점을 위해 필요한 것"이 명시되어 다음 스텝이 명확

### 5-7. 독자에 맞춰 포맷을 전환하라 (v4 교훈)

같은 내용이라도 **독자에 따라 최적의 포맷이 다르다.** v3까지의 전략 문서는 전략팀 내부 검토에 최적화되어 있었지만, 1차 조직장 리뷰에서는 "어려운 용어"와 "맥락 없는 숫자"가 장벽이 되었다. v4의 포맷 전환은 내용을 바꾸지 않고 전달 방식만 바꿔 의사결정 속도를 높인다.

핵심 포인트:
- **Before/After 예시를 프롬프트에 포함하라:** 추상적 지시보다 구체적 변환 예시가 효과적
- **회의 어젠다 형태를 활용하라:** "오늘 결정할 것 3가지"는 회의 참석자의 집중을 유도
- **[논의 포인트] 삽입:** 독자가 수동적으로 읽는 것이 아니라 능동적으로 참여하게 만듦

## 6. 사용된 에이전트/스킬 목록

### 에이전트 역할 요약

| 에이전트 | 역할 | 스킬 |
|---------|------|------|
| **research-agent** | 주제별 웹 리서치 수행. 검색 키워드 기반으로 데이터를 수집하고 구조화된 리서치 문서를 산출 | report-research |
| **outline-agent** | 리서치 결과를 기반으로 보고서의 목차, 구조, 섹션별 핵심 메시지를 설계 | report-outline |
| **writer-agent** | 구조 설계안과 리서치 자료를 기반으로 본문 초안을 작성 | report-draft |
| **fact-check-agent** | 초안의 모든 수치와 주장을 리서치 원본과 대조 검증 | (내장 검증 로직) |
| **editor-agent** | 문체 통일, 흐름 개선, 포맷 정리, 최종 품질 관리 | report-edit |

### 하네스(harness) 스킬의 역할

하네스는 에이전트 팀의 **오케스트레이터**다. 다음을 수행한다:

1. **에이전트 팀 구성:** 작업에 필요한 에이전트들을 정의하고 각각에 스킬을 배정
2. **실행 순서 관리:** 팬아웃(병렬)/파이프라인(순차) 실행 흐름을 제어
3. **파일 경로 연결:** 각 에이전트의 출력을 다음 에이전트의 입력으로 전달
4. **품질 게이트:** 각 단계 완료 후 산출물 품질을 확인하고 다음 단계 진행 여부 판단

실제로 이 프로젝트에서는 `report-orchestrator` 스킬이 하네스 역할을 수행했다:
- 리서치 3개 에이전트를 병렬 실행 (팬아웃)
- 결과를 병합하여 outline → writer → fact-check → editor 파이프라인 실행
- 각 단계의 파일을 `_workspace/` 폴더에 순서대로 저장

## 7. 재현 가이드

### 7-1. 이 프로세스를 다른 주제로 재현하려면?

이 워크플로우는 **네이버 스포츠 마이팀**에 특화된 것이 아니라, 모든 전략 보고서에 적용 가능한 범용 프로세스다.

### 7-2. 변경해야 할 부분

| 항목 | 예시 (이 프로젝트) | 다른 프로젝트로 변경 |
|------|-------------------|-------------------|
| **주제** | 네이버 스포츠 마이팀 | (새 주제) |
| **리서치 영역 3개** | 국내 현황, 글로벌 벤치마크, 구독/팬덤 전략 | 주제에 맞게 3개 영역 재정의 |
| **피드백 페르소나** | CPM (서비스기획 최고 책임자) | 실제 리뷰어의 역할/관점 |
| **보강 리서치 타겟** | VOC, 경쟁 위협, 실행 설계 | 피드백에서 도출된 약점 영역 |
| **루브릭 평가 축** | 유저 인사이트, 데이터 근거 등 8개 | 보고서 목적에 맞는 평가 기준 |
| **v4 포맷** | 조직장 리뷰 회의 자료 | 실제 발표/리뷰 대상에 맞는 포맷 |

### 7-3. 변경하지 않아도 되는 부분

다음은 어떤 주제에서든 동일하게 적용 가능한 **범용 패턴**이다:

1. **4-Phase 반복 개선 구조:** v1(빠른 초안) → v2(피드백 반영) → v3(타겟 보강) → v4(포맷 전환)
2. **팬아웃/팬인 + 파이프라인 아키텍처:** 리서치 병렬 → 파이프라인 순차
3. **파이프라인 4단계:** outline → writer → fact-check → editor
4. **피드백 시뮬레이션 패턴:** 페르소나 설정 → 비판적 검토 → 개선 방향 도출
5. **검색 키워드 명시:** 리서치 프롬프트에 구체적 검색 쿼리 포함
6. **[검증 필요] 태깅:** 확인 불가 정보는 명시적으로 표시
7. **루브릭 자체 평가:** 보고서 말미에 평가 매트릭스 포함
8. **두괄식 구조:** 섹션 첫 문장 = 핵심 메시지
9. **포맷 전환:** 같은 내용을 독자/상황에 맞게 재구성

### 7-4. 실행 체크리스트

```
□ 1. 요구사항 정의: 보고서 목적, 독자, 분량 결정
□ 2. 리서치 영역 3개 정의: 주제에 맞는 조사 영역 분류
□ 3. 리서치 에이전트 3개 병렬 실행: 검색 키워드 명시
□ 4. 리서치 병합 (팬인)
□ 5. 파이프라인 실행: outline → writer → fact-check → editor
□ 6. v1 산출
□ 7. 피드백 시뮬레이션: 실제 리뷰어 페르소나로 비판적 검토
□ 8. 보강 리서치 병렬 실행: 피드백에서 도출된 약점 영역
□ 9. v2 재작성
□ 10. 루브릭 갭 분석: 각 평가 축별 현재 점수 vs 목표
□ 11. 타겟 리서치 병렬 실행: 갭을 메울 데이터 수집
□ 12. v3 최종 재작성: 보강 리서치 + 구조 재정의
□ 13. 루브릭 자체 평가 포함
□ 14. 독자/상황에 맞는 포맷 전환 (v4): 회의 자료, 발표 자료 등
```
